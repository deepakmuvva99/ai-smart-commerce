PART 1: EXECUTIVE SUMMARY & SYSTEM PURPOSE
--------------------------------------------------------------------------------
The Smart Commerce Platform is a next-generation, autonomous e-commerce engine designed to eliminate the need for human analysts in pricing strategies. By leveraging state-of-the-art Deep Learning (DL) and Reinforcement Learning (RL), the system dynamically calculates the optimal price for every product in real-time, maximizing revenue while preserving inventory constraints.

Unlike traditional rule-based dynamic pricing (e.g., "drop price by 10% if inventory > 100"), this system uses a dual-engine AI:
1. A Deep Learning model (BiLSTM) to forecast exact future demand.
2. A Reinforcement Learning model (Soft Actor-Critic) to manipulate prices based on that predicted demand and the current market state.

The application is built using a modern microservices architecture, fully containerized via Docker, and designed to run safely behind an Nginx reverse proxy on an Azure Virtual Machine.

--------------------------------------------------------------------------------
PART 2: HIGH-LEVEL ARCHITECTURE OVERVIEW
--------------------------------------------------------------------------------
The system is divided into three primary, isolated microservices. This separation of concerns allows each piece to scale independently, fail gracefully, and securely communicate over internal Docker networks.

1. The Frontend Proxy (React + Vite + Nginx)
2. The Core Backend (Python FastAPI)
3. The AI Engine (Python + PyTorch + LightGBM)

Flow of a user request:
- A user visits the public IP address of the Azure VM.
- The request hits Port 80, which is mapped to the Nginx reverse proxy container.
- If the user is requesting a static UI file (HTML/CSS/JS), Nginx serves the compiled React Vite build directly from `/usr/share/nginx/html`.
- If the user clicks a button that triggers an API call (e.g., `/api/products`), Nginx intercepts this via the `location /api/` block and proxies the request backward into the hidden internal Docker network, forwarding it to the Core Backend on Port 8000.
- The Core Backend processes the request. If it needs a new dynamic price, it internally pings the AI Engine container via HTTP on Port 8001. 
- The AI Engine computes the price, returns it to the backend, the backend saves it to Azure MySQL, and then returns the JSON payload back through Nginx to the user's browser.

PART 3: THE FRONTEND APPLICATION (REACT + VITE)
--------------------------------------------------------------------------------
The frontend is built for extreme speed and a beautiful user experience. It serves as both the consumer-facing shopping interface and the internal administration dashboard.

1. Technology Stack
-------------------
- React.js: Component-based UI library.
- Vite: Next-generation build tool offering instantaneous hot-module replacement and highly optimized production builds.
- Axios: For handling asynchronous HTTP requests to the backend.
- Context API: Used for global state management (specifically for tracking Authentication tokens and User Roles across pages).

2. User Module
--------------
The standard user workflow contains everything expected of a modern e-commerce platform:
- Registration & Login: Users create accounts. Passwords are minimum 8 characters to prevent brute-forcing. Upon login, a JWT is generated by the backend and stored locally by the frontend Context.
- Product Catalog: Displays all available items. When a user clicks an item, the frontend requests data. Nginx routes this to the backend. This click also triggers a hidden "Traffic Log" event, sending telemetry data to the backend so the AI knows which items are trending.
- Cart & Checkout: Standard shopping flow, heavily secured by backend validations.

3. Admin Dashboard
------------------
The system differentiates roles via the JWT payload. If the decoded token identifies the user as an `admin`, the frontend renders the hidden `/admin` routes.
- Inventory Management: Admins can manually add products, adjust base/cost prices, and add variants (sizes/colors).
- Machine Learning Analytics: This is the crown jewel of the UI. It features real-time charts plotting `Predicted Demand` vs `Actual Revenue` vs `Current Price Multiplier`. Admins can watch the Reinforcement Learning agent make decisions live.

4. Nginx Reverse Proxy Integration
----------------------------------
To solve Cross-Origin Resource Sharing (CORS) issues and provide a single entry point, the frontend is served by Nginx. The Dockerfile utilizes a multi-stage build:
Stage 1 (Node): Installs dependencies, runs `npm run build`, and generates optimized static files.
Stage 2 (Nginx): Copies the compiled files to `/usr/share/nginx/html` and applies `smart-commerce.conf`.

The `smart-commerce.conf` is crucial:
```nginx
server {
    listen 80;
    
    # Serves the React UI
    location / {
        root /usr/share/nginx/html;
        try_files $uri $uri/ /index.html;
    }
    
    # Bridges the Frontend to the Backend
    location /api/ {
        proxy_pass http://smart_commerce_backend:8000/;
    }
}
```

PART 4: THE CORE BACKEND (PYTHON FASTAPI)
--------------------------------------------------------------------------------
The backend is the nervous system of the platform. It handles business logic, database transactions, security, and orchestrates the AI.

1. Technology Stack
-------------------
- FastAPI: An incredibly fast, modern web framework based on standard Python type hints. Selected over Django/Flask because of its asynchronous capabilities, which are essential for waiting on heavy AI computations without blocking other users.
- Uvicorn: The ASGI server powering FastAPI.
- SQLAlchemy: The Object Relational Mapper (ORM) translating Python objects to SQL queries.
- Pydantic: Strict data validation schemas.
- Bcrypt & JWT: For hashing passwords and issuing secure tokens.

2. Database Architecture (Azure MySQL integration)
--------------------------------------------------
The platform was recently upgraded from a local SQLite database to a cloud-hosted Azure MySQL Flexible Server to support high availability.
The SQLAlchemy engine connects using the credentials stored securely in the `.env` file:
`DB_USER`, `DB_PASSWORD`, `DB_HOST`, etc.

Key Tables:
- Users: Stores email, hashed password, role (admin/user).
- Products & Variants: Stores items. Critically, it tracks `base_price` (the lowest we can sell for) and `cost_price` (manufacturing cost).
- Traffic Logs: Every UI click inserts a row here. This is the raw dataset the AI uses to predict trends.
- AI Metrics: Stores the historical tracking of the AI agent's actions for the dashboard charts.

3. Background Schedulers & AI Integration
-----------------------------------------
The backend doesn't just wait for API requests; it runs background threads.
- The Price Update Loop: Every few minutes, a background task loops through all products. It aggregates the recent traffic logs and sales history, and packages this state.
- It makes an HTTP request to the internal AI Engine (`http://ai_engine:8001/predict`).
- The AI responds with a new price multiplier (e.g., 1.05x).
- The Backend applies this multiplier to the `base_price`, updates the database, and the new price instantly reflects on the frontend.
- The Reward Loop: Later, the backend calculates how much profit was generated during that timeframe, and sends a `/reward` payload back to the AI, allowing the AI to learn if its pricing decision was good or bad.

--------------------------------------------------------------------------------
PART 5: THE AI ENGINE (PREDICTION & REINFORCEMENT LEARNING)
--------------------------------------------------------------------------------
This is a completely separate microservice. It runs its own Uvicorn server and is dedicated purely to massive mathematical computations. It handles the dual-engine AI approach.

Phase 1: Deep Learning (DL) - The BiLSTM Predictor
--------------------------------------------------
Goal: Predict exact sales volume over the next 24 hours.
Why BiLSTM? (Bidirectional Long Short-Term Memory)
Standard LSTMs read data in one direction (past to future). In e-commerce, context exists in both directions. A massive spike on Sunday might explain a dip on Monday. BiLSTMs process the timeline forwards and backwards simultaneously, capturing complex, hidden seasonal trends.
Input: The backend sends an array of historical traffic logs, cart additions, and past sales.
Output: A raw float representing `predicted_demand` (e.g., 14.5 items).

Phase 2: Reinforcement Learning (RL) - Soft Actor-Critic (SAC)
--------------------------------------------------------------
Goal: Decide the optimal price.
Why SAC? Soft Actor-Critic is a state-of-the-art RL algorithm designed for continuous action spaces. We aren't choosing between "Price A" and "Price B". We are choosing an exact multiplier on a sliding scale (e.g., 1.00x to 2.50x). 
Furthermore, SAC maximizes an "entropy" term alongside the standard reward. This means it actively tries to explore new, random pricing strategies occasionally, preventing it from getting stuck using the same boring price forever.

The RL Environment Formulation:
- State Space: The AI looks at: [Current Inventory, Traffic Volume, Competitor Pricing (if available), and crucially, the DL's Predicted Demand].
- Action Space: A continuous float representing the `price_multiplier`.
- Reward Function: The system is rewarded based on PROFIT, not just revenue. 
  Reward = (Final Price - Cost Price) * Items Sold.
  If the AI prices too high, Items Sold drops to 0. Reward = 0.
  If the AI prices too low, Final Price approaches Cost Price. Reward = 0.
  The AI must find the perfect equilibrium.

PART 6: DEPLOYMENT & DEVOPS (DOCKER & AZURE)
--------------------------------------------------------------------------------
The entire platform is built for portability using Docker Compose.

1. Docker Networking
--------------------
We use a two-network topology for security:
- `internal_network`: A completely isolated virtual network. The AI Engine and Backend live here. They can talk to each other, but the outside internet cannot reach them. If a hacker scans the VM's IP, ports 8000 and 8001 simply do not exist.
- `proxy_network`: Bridges the Nginx frontend to the Backend. Nginx straddles the outside world (Port 80/443) and the internal network.

2. The Azure VM Deployment Strategy
-----------------------------------
The platform runs on an Ubuntu 24.04 LTS Azure Virtual Machine (`Standard_B2ls_v2` with 2 vCPUs and 4GB RAM, sufficient for running the PyTorch and FastAPI workloads).

The Deployment Workflow:
1. Code is pushed to Git.
2. The VM operator SSHs into the server using an Ed25519 or RSA private `.pem` key.
3. The repository is cloned.
4. A custom `azure_vm_setup.sh` script automatically provisions the VM, installing Docker Engine, Docker Compose plugin, and configuring user groups.
5. Environment variables (`.env`) are securely created on the VM, pointing the backend to the Azure MySQL Flexible server. Crucially, the frontend `.env` is updated to set `VITE_API_URL=/api`, ensuring the browser routes requests through the Nginx proxy rather than attempting to hit a non-existent local port.
6. The operator runs `docker compose up -d --build`.
7. Docker pulls the base images (`python:3.10-slim` for backend/AI, `node:20-alpine` and `nginx:alpine` for frontend).
8. The images are built locally on the VM, tagging them properly (`smart_commerce_backend_image`, etc.) to avoid `<none>` anonymous tags.
9. Containers boot, attach to the custom networks, and the platform becomes live.

PART 7: END-TO-END WORKFLOW EXAMPLE (DYNAMIC PRICING IN ACTION)
--------------------------------------------------------------------------------
To truly understand the system, let us trace a single product: A "Smart Watch".
1. Admin logs into the dashboard and creates the "Smart Watch" with a Base Price of $100 and Cost Price of $60.
2. 50 users visit the site and click on the Smart Watch.
3. The React frontend sends 50 API requests to Nginx -> Backend to log this traffic.
4. The Backend background thread wakes up. It gathers the 50 clicks, current inventory, and sends it to the AI Engine.
5. The AI Engine's BiLSTM looks at the 50 clicks and predicts: "Demand will be high today, expecting 12 sales."
6. The AI Engine's SAC RL agent looks at the state [Demand: 12, Inventory: 100]. It decides to exploit this high demand. It outputs an action: `price_multiplier = 1.25`.
7. The Backend receives `1.25`, updates the Azure MySQL database. The Smart Watch is now $125.
8. A user buys the watch.
9. An hour later, the Backend calculates the profit: ($125 - $60) * 1 sale = $65 profit.
10. The Backend sends a Reward of +65 to the AI Engine.
11. The AI Engine updates its neural network weights, learning that raising prices during high traffic yields excellent rewards.

This autonomous cycle repeats infinitely across thousands of products, turning the Smart Commerce Platform into a self-optimizing, highly profitable e-commerce machine.

PART 8: COMPLETE DATAFLOW & WORKFLOW TRIGGERS
--------------------------------------------------------------------------------

To fully visualize how data moves through the microservices, here is a detailed breakdown of the exact triggers for both Customers and Administrators.

=========================================
A. THE CUSTOMER WORKFLOW (DATAFLOW)
=========================================

1. Browsing & Product Discovery
   - Trigger: Customer opens the web browser to the public IP.
   - Flow: Browser -> Nginx (Port 80) -> Serves static React UI.
   - Action: React fetches product catalog. Nginx proxies request to Backend (Port 8000). Backend reads Azure MySQL and returns the product list.

2. Interaction & Telemetry Generation (The most important step for AI)
   - Trigger: Customer clicks on a specific product to view its details.
   - Flow: React sends a hidden HTTP POST request to `/api/traffic_log`.
   - Action: Backend receives this, timestamps it, and logs `event_type=view`, `product_id`, and `user_id` into the Azure MySQL database.

3. Purchase & Inventory Mutation
   - Trigger: Customer clicks "Add to Cart" and checks out.
   - Flow: React POSTs to `/api/orders`.
   - Action: Backend validates the JWT token. It deducts the purchased quantity from the `product_variants` inventory table. It generates a final Order record.

4. The Autonomous Price Shift (Hidden from Customer)
   - Trigger: Every 5 minutes, a cron job / background thread runs on the Backend.
   - Flow: Backend -> Queries recent traffic logs -> Packages state -> Sends HTTP POST to AI Engine -> AI replies with `price_multiplier` -> Backend updates DB.
   - Action: When the Customer refreshes the page, the item is now a different price!

PART 9: ADMIN ANALYTICS DASHBOARD, CHARTS & AI CONTROLS
--------------------------------------------------------------------------------

The Admin Dashboard is the central command center for the entire platform. While the AI is designed to operate autonomously, human administrators need deep visibility into exactly *why* the AI is making certain pricing decisions. 

=========================================
A. HOW THE CHARTS WORK (TECHNOLOGY)
=========================================
The graphs on the Admin Dashboard are rendered on the React frontend using a modern charting library (e.g., Chart.js or Recharts). 
- Data Polling: The React application polls the FastAPI backend endpoint `/api/admin/metrics` periodically.
- Data Source: The backend queries the `ai_metrics` and `price_history` tables in the Azure MySQL database to return time-series data.
- Aggregation: Because the AI can make hundreds of decisions an hour, the backend aggregates this data into readable chunks (e.g., hourly averages) before sending it to the frontend to prevent browser lag.

=========================================
B. WHAT THE CHARTS DISPLAY
=========================================
There are three primary visualizations an Admin will monitor:

1. Predicted Demand vs. Actual Sales (The BiLSTM Tracker)
   - What it shows: A multi-line graph comparing the AI's predicted sales volume against the real recorded purchases.
   - Purpose: If the "Predicted" line heavily diverges from the "Actual" line, it signals to the Admin that consumer trends have drastically changed (e.g., a sudden viral social media post) and the Deep Learning model may need a manual retraining trigger.

2. Price Multiplier Fluctuations (The RL Tracker)
   - What it shows: A timeline of the `price_multiplier` value chosen by the SAC Reinforcement Learning Agent mapped alongside current Inventory levels.
   - Purpose: Admins can watch the AI dynamically raise the price (e.g., 1.5x) as inventory drops to scarcity levels, ensuring profit margins are maximized before stock runs out.

3. Cumulative Profit / Reward Mapping
   - What it shows: A bar chart displaying the raw profit generated over time, specifically highlighting the extra margin generated by the dynamic pricing compared to a static baseline price.

=========================================
C. HOW TO CONTROL THE SYSTEM
=========================================
Administrators are not locked out of the AI's decisions. They have several manual control levers built into the dashboard:

1. Setting Hard Bounds (Price Ceilings/Floors)
   - Admins can enforce boundaries to protect brand reputation. For example, setting a `max_price_multiplier` of 2.0x prevents the AI from ever doubling the price of an item, even in extreme scarcity, to avoid price-gouging complaints.

2. Manual Price Overrides
   - If an Admin wants to run a promotional holiday sale, they can toggle an "AI Override" switch for specific products, locking the price to a fixed discount and ignoring the AI's recommendations until the toggle is flipped back.

3. Inventory Injections
   - As explained in the workflow, typing a new stock value (e.g., adding +1000 units) into the Admin UI instantly resets the AI's scarcity parameters, forcing it to drop prices to clear the massive new volume.
